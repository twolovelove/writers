### RLS 체크리스트 — Supabase에 새 테이블을 추가할 때마다
1. `alter table <table> enable row level security;`를 빠뜨리지 않았는가?
2. 정책이 "본인 데이터만" 접근 가능하게 `auth.uid() = user_id` 조건을 걸고 있는가? (전체 공개가 목적이 아니라면)
3. 관리자 전용 열람이 필요하면 이메일 하드코딩이 아니라 `admins` 테이블을 참조하는가? (`reviews` 테이블 정책 참고)
4. 로컬에서 다른 계정 두 개로 로그인해 "내 데이터만 보이고 남의 데이터는 안 보이는지" 직접 확인했는가? (RLS는 서버 쪽 격리만 보장 — 클라이언트가 이 데이터를 LocalStorage 등 계정과 무관한 곳에 캐시해두면 이 격리가 클라이언트 쪽에서 깨질 수 있다는 걸 별도로 체크할 것. 2026-07-06에 `entries`가 이 문제로 실제 크리티컬 버그였음 — 2-3 참고)
5. `supabase/schema.sql`에 반영한 뒤, README의 "Supabase 설정" 안내대로 실제 프로젝트 SQL 편집기에도 실행했는가?
6. 새 테이블에 `updated_at` 같은 "최신 여부 판단" 컬럼이 있다면, 클라이언트 기기 시계가 아니라 서버(`now()`, 트리거)가 채우게 했는가? (기기 시계가 어긋나면 최신 데이터가 오래된 데이터에 덮어써질 수 있음)

## 2. 회사 관점 4대 유지보수 영역 — 우선순위

**우선순위 기준**
- 🔴 **P0 — 지금 필요**: 리스크가 실현되면 되돌리기 어렵거나(법적/데이터), 이미 손댄 기능이라 방치하면 바로 문제가 됨
- 🟠 **P1 — 가까운 시일**: 지금 만들어두면 이후가 훨씬 쉬워지고, 비용이 작음 (유저/매출 생기기 *전에* 준비)
- 🟡 **P2 — 트래픽/매출이 생기면**: 지금 만들어도 검증할 유저가 없어 낭비가 될 수 있음
- ⚪ **보류**: 지금 단계에서는 실행 안 함(재검토 시점만 기록)

   - **확정 짓고 싶으면**: 실제 배포 앱을 로그아웃 상태 시크릿창에서 열고 콘솔에 `supabase.from('reviews').select('*')`를 직접 실행 — 빈 배열이면 확정적으로 안전.
   - **다음에 만지게 되면**: 위 `schema.sql` 변경을 라이브 SQL 편집기에 재실행할 것(정책 자체엔 문제 없었지만 방어적으로 넣어둔 거라 급하진 않음).

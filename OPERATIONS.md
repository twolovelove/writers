# 운영 구조 & 유지보수 계획

> **문서 목적**: "매일 1,000자 글쓰기"를 하나의 서비스로 보고, 개발/운영/디자인 역할별 관리 포인트와 회사 관점의 유지보수 항목을 한 곳에 정리한다. 실제로는 1인이 모든 역할을 겸하고 있으므로, 아래 "팀"은 조직이 아니라 **역할(모자)** 구분으로 읽는다.
>
> **현재 단계**: 실사용자 거의 없음 / 매출 없음 / 배포·마케팅 이전 단계. 우선순위는 이 단계를 기준으로 매겼고, 유저·매출이 생기면 재조정이 필요하다 (아래 3번 로드맵 참고).
>
> **최종 업데이트**: 2026-07-06 (P0·P1 전 항목 완료, 기술 부채 전 항목 해소 — "에러 모니터링 없음"까지 마무리. P2는 아직 착수하지 않고 메모로만 남겨둠 — 유저/매출이 생기면 재검토)

## 1. 역할별 기능 & 관리 포인트

| 기능 영역 | 개발 (Dev) | 운영 (Ops) | 디자인 (Design) |
| --- | --- | --- | --- |
| **글쓰기 에디터**<br>(`Editor.tsx`, `EditorToolbar`) | 자동저장 디바운스, 서식 적용 로직(`textFormat.ts`) 정상 동작 유지 | 하루 1,000자 잠금 규칙(`useDailyLock`)이 기획 의도대로 동작하는지 주기 점검 | 원고지 느낌의 몰입감, 진행률 표시(`ProgressBar`)의 심리적 동기부여 |
| **첨삭 노트**<br>(`feedback.ts`, `FeedbackPanel`) | 규칙 기반 로직 — `feedback.test.ts`에 회귀 테스트 있음, 새 규칙 추가 시 테스트도 같이 갱신 | 첨삭 문구가 유저에게 상처 주지 않는 톤인지 주기적 재검토 | 칭찬/제안/총평 구조가 읽기 편한지, 정보 위계 |
| **아카이브 / 모음집**<br>(`ArchiveCalendar`, `Compilation`) | 인쇄/PDF 저장 로직 유지, 월별 데이터 집계 정확성 | 이 기능이 사실상 "데이터 내보내기" 대체 수단임을 인지(3-3 참고) | 달력·목차·인쇄본 레이아웃이 실제로 "책처럼" 보이는지 |
| **로그인 & 동기화**<br>(`useSession`, `useEntrySync`, `lib/entries.ts`) | RLS 정책 유지, 새 테이블 추가 시 아래 "RLS 체크리스트" 준수 | 동기화 실패는 이제 Sentry(`captureError`)로 잡힘 — DSN 설정 후 실제 발생 빈도 주기 확인 | 로그인 유도 문구, 동기화 상태를 유저가 안심할 수 있게 노출 |
| **리뷰 위젯 & 관리자 페이지**<br>(`ReviewWidget`, `AdminReviews`) | 관리자 판별은 `admins` 테이블 단일 기준(`useIsAdmin.ts`) — 새 관리자 추가 시 이메일 하드코딩 대신 이 테이블에 행 추가 | 들어온 리뷰를 실제로 정기적으로 확인·반영하는 루틴 | 위젯이 방해되지 않으면서 눈에 띄는 위치/톤 |
| **설정 / 개인정보 / 탈퇴**<br>(`Settings.tsx`, `PrivacyPolicy.tsx`) | 탈퇴 시 로컬+서버 데이터 모두 삭제되는지(`deleteAllEntries`) 회귀 확인 | 개인정보처리방침 문구가 실제 데이터 처리 방식과 항상 일치하는지(기능 바뀔 때마다 동반 수정) | 탈퇴/삭제 같은 되돌릴 수 없는 액션의 경고 UX |

### 개발이 별도로 계속 관리해야 할 기술 부채
- ~~`supabase/schema.sql` 미적용~~: 확인 결과 이미 라이브 Supabase 프로젝트에 적용되어 있음(2026-07-06 확인). `entries`/`admins`/`reviews` 테이블과 RLS 모두 동작 중.
- ~~`ADMIN_EMAIL` 중복~~: 확인 결과 이미 해소됨 — 관리자 판별은 `admins` 테이블(`schema.sql`) 단일 기준이고 `config.ts`의 `SUPPORT_EMAIL`은 문의처 안내용으로만 쓰임 (`src/hooks/useIsAdmin.ts` 참고).
- ~~CI/CD 없음~~: `.github/workflows/ci.yml` 추가(2026-07-06). `main` push·PR마다 lint+타입체크+빌드 자동 실행. 아직 배포(호스팅) 자체는 연결 안 함 — 배포처가 정해지면 CD 단계 추가 필요.
- ~~테스트 없음~~: Vitest 도입(2026-07-06). `feedback.test.ts`(첨삭 규칙 10개 케이스), `archive.test.ts`(정렬·병합·스트릭 등 유닛 테스트) 추가, CI에 `npm run test` 단계 연결. 새 규칙/로직 추가 시 테스트도 같이 갱신할 것.
- ~~에러 모니터링 없음~~: `@sentry/react` 도입(2026-07-06). `src/lib/monitoring.ts`가 `VITE_SENTRY_DSN` 설정 시에만 초기화, `main.tsx`에서 앱 전체를 `ErrorBoundary`(`src/components/ErrorFallback.tsx`)로 감싸 렌더링 중 에러를 잡고, 조용히 흡수되던 `pushEntry`/`useEntrySync` 동기화 실패도 `captureError`로 전송하도록 연결. DSN 미설정 시 콘솔 경고만 남기고 앱 동작에는 영향 없음. **Sentry 프로젝트 생성·DSN 발급은 지금 미룸** — 아직 배포 전(로컬만 사용)이라 "프로덕션에서 유저가 겪는 에러를 모른다"는 리스크 자체가 실현될 상황이 아님. 배포 시점에 GA4 측정 ID와 함께 챙기면 됨(2026-07-06 결정).

## 2. 회사 관점 4대 유지보수 영역 — 우선순위

**우선순위 기준**
- 🔴 **P0 — 지금 필요**: 리스크가 실현되면 되돌리기 어렵거나(법적/데이터), 이미 손댄 기능이라 방치하면 바로 문제가 됨
- 🟠 **P1 — 가까운 시일**: 지금 만들어두면 이후가 훨씬 쉬워지고, 비용이 작음 (유저/매출 생기기 *전에* 준비)
- 🟡 **P2 — 트래픽/매출이 생기면**: 지금 만들어도 검증할 유저가 없어 낭비가 될 수 있음
- ⚪ **보류**: 지금 단계에서는 실행 안 함(재검토 시점만 기록)

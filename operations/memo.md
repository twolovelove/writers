두 가지 니즈가 섞여 있어 분리해서 본다: (a) 유저가 첨삭 없이 일기만 쓰고 싶은 UX 취향, (b) AI 첨삭 API 토큰 비용이 실제로 부담될 때의 비용 통제.

- **(a) 유저별 토글 → `Settings.tsx`에 넣는 방향으로 결정.** 첨삭 로직 자체(`feedback.ts`)는 이미 규칙 기반이라 서버 API 호출 비용이 없고(생성형 AI 호출 아님), 구현 비용이 작다. 유저가 켜고 끌 수 있게 하는 게 UX상 자연스러움. 실사용자가 생겨 "끄고 싶다"는 신호가 확인되면 P1로 올려 구현.
- **(b) 관리자 전역 토글(비용 통제용) → P2로 보류.** 지금 첨삭이 규칙 기반이라 토큰 비용 자체가 발생하지 않으므로, 이 항목은 "첨삭을 생성형 AI로 고도화"(수익 모델 항목, [로드맵.md](./로드맵.md) 참고)하는 시점에 실제로 필요해짐. 그 전에 관리자 킬스위치부터 만드는 건 아직 존재하지 않는 비용을 통제하는 것이라 순서가 이름.

**다음 액션**: 지금 당장 구현하지 않음. (a)는 유저 요청이 들어오면, (b)는 첨삭을 생성형 AI로 바꾸는 시점에 각각 재검토.

   - **원인**: `useDraft.ts`의 자동저장이 제목·본문이 둘 다 빈 상태에서도 디바운스 타이머가 한 번 더 돌 때 그대로 Supabase에 upsert해버려, 글감만 열어보고 바로 나가도 `entries`에 0자짜리 행이 생겼음.

   - **번복(2026-07-08)**: 이 결론은 "완료 못 한 카테고리의 초고는 하루 1회 완료 잠금 때문에 영영 이어 쓸 수 없다"는 사실이 다시 드러나며 뒤집힘 — 여러 카테고리를 둘러보는 게 장점이 되려면 나중에 재개할 수 있어야 하는데, 실제로는 재개가 불가능해 그냥 죽은 초고만 쌓였음. 유저가 카테고리를 고르는 구조 자체를 없애고 하루 하나만 자동 추천하는 방식으로 개편(위 업데이트 로그 2026-07-08 참고).
   - **주의**: 이미 Supabase에 저장된 기존 0자 유령 행(스크린샷의 "오늘의 낙서장" 등)은 이 수정으로 자동 삭제되지 않음 — 필요하면 Supabase 테이블 편집기에서 `char_count = 0`이고 `title`도 비어있는 행을 수동으로 지울 것.

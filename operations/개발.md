### RLS 체크리스트 — Supabase에 새 테이블을 추가할 때마다
1. `alter table <table> enable row level security;`를 빠뜨리지 않았는가?
2. 정책이 "본인 데이터만" 접근 가능하게 `auth.uid() = user_id` 조건을 걸고 있는가? (전체 공개가 목적이 아니라면)
3. 관리자 전용 열람이 필요하면 이메일 하드코딩이 아니라 `admins` 테이블을 참조하는가? (`reviews` 테이블 정책 참고)
4. 로컬에서 다른 계정 두 개로 로그인해 "내 데이터만 보이고 남의 데이터는 안 보이는지" 직접 확인했는가? (RLS는 서버 쪽 격리만 보장 — 클라이언트가 이 데이터를 LocalStorage 등 계정과 무관한 곳에 캐시해두면 이 격리가 클라이언트 쪽에서 깨질 수 있다는 걸 별도로 체크할 것. 2026-07-06에 `entries`가 이 문제로 실제 크리티컬 버그였음 — 아래 업데이트 로그 참고)
5. `supabase/schema.sql`에 반영한 뒤, README의 "Supabase 설정" 안내대로 실제 프로젝트 SQL 편집기에도 실행했는가?
6. 새 테이블에 `updated_at` 같은 "최신 여부 판단" 컬럼이 있다면, 클라이언트 기기 시계가 아니라 서버(`now()`, 트리거)가 채우게 했는가? (기기 시계가 어긋나면 최신 데이터가 오래된 데이터에 덮어써질 수 있음)

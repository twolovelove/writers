# 로드맵 & 회사 관점 유지보수 (비즈니스 · 법률 · 전체 우선순위)

> [operations/INDEX.md](./INDEX.md)에서 전체 문서 구조를 먼저 확인하세요.

**우선순위 기준**
- 🔴 **P0 — 지금 필요**: 리스크가 실현되면 되돌리기 어렵거나(법적/데이터), 이미 손댄 기능이라 방치하면 바로 문제가 됨
- 🟠 **P1 — 가까운 시일**: 지금 만들어두면 이후가 훨씬 쉬워지고, 비용이 작음 (유저/매출 생기기 *전에* 준비)
- 🟡 **P2 — 트래픽/매출이 생기면**: 지금 만들어도 검증할 유저가 없어 낭비가 될 수 있음
- ⚪ **보류**: 지금 단계에서는 실행 안 함(재검토 시점만 기록)

| 탈퇴 시 `auth.users` 계정 자체의 완전 삭제(진짜 하드 딜리트) | ✅ 배포·실사용자 검증 완료(2026-07-09) | `supabase/functions/delete-account`(Edge Function) 추가 — 요청자 JWT로 본인 확인 후 서비스 롤 키로 `auth.admin.deleteUser()` 호출. `schema.sql`에 `entries`/`reviews`/`admins` 세 테이블의 `user_id` FK에 `on delete cascade` 추가해 계정 삭제 시 관련 행도 함께 정리되게 함. `Settings.tsx`의 `handleWithdraw`가 이 함수를 호출하고, 실패 시에만 기존 `deleteAllEntries`로 폴백 | 2026-07-09 일반 유저 계정으로 실제 탈퇴 테스트 완료 — 계정 삭제와 작성한 글 삭제 모두 정상 동작 확인 |
| 🔴 계정 간 글 데이터 혼입 (LocalStorage 캐시) | ✅ 수정 완료(2026-07-06) | 전체 코드 리뷰에서 발견: `entries`가 `writer:draft:{date}:{category}` 키로 LocalStorage에도 캐시됐는데, 이 키에 계정 구분이 없고 로그아웃도 LocalStorage를 지우지 않았음. 같은 기기(가족 공용 PC 등)에서 A 계정으로 로그아웃 후 B 계정으로 로그인하면, B가 A의 글을 그대로 보고 동기화 로직이 그 글을 B의 Supabase 계정에 영구 반영해버리는 경로가 있었음 — "본인 계정으로 로그인했을 때만 열람할 수 있고 다른 누구도 볼 수 없다"는 개인정보 안내 문구와 실제 동작이 어긋났던 크리티컬 버그 | LocalStorage 캐시/병합 로직(`useEntrySync`, `getAllDrafts`, `applyNewerRemoteEntry`)을 통째로 제거하고 `entries`가 Supabase만을 유일한 저장소로 쓰도록 재구조화(`useEntries`가 로그인 시 1회 불러와 화면 간 공유 캐시로만 들고 있음, 계정이 바뀌면 자동으로 다시 불러옴). 부수 효과로 클라이언트 시계를 신뢰하던 last-write-wins 병합 로직 자체가 사라짐(아래 항목과 함께 해결) |
| 🟡 동기화 충돌 해결이 기기 시계를 신뢰함 | ✅ 수정 완료(2026-07-06) | 기존엔 `updatedAt`을 클라이언트 기기 시계로 채워 "더 최신" 판단 기준으로 썼음 — 기기 시계가 어긋나 있으면 오래된 기기의 저장이 조용히 최신 저장을 덮어쓸 수 있었음(유저에게 아무 경고 없이) | 위 크리티컬 버그 수정과 함께 클라이언트 병합 로직 자체를 없앴고, `schema.sql`에 `entries_set_updated_at` 트리거를 추가해 `updated_at`을 항상 서버(`now()`)가 채우도록 함. **트리거 반영·2026-07-07 RLS 정책 방어 블록 모두 라이브 SQL 편집기에 재실행 완료** |
